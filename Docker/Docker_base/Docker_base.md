## Вопрос 2  
В чем заключается разница между командами docker run и docker start?  

- docker run создает новый контейнер и запускает его с заданными параметрами, а docker start перезапускает контейнер с сохраненными настройками  
- docker run создает временный контейнер, который удаляется после завершения выполнения, а docker start запускает контейнер, сохраняя его состояние  
- docker run создает и запускает новый контейнер, а docker start перезапускает существующий контейнер, если он был остановлен  
- docker run запускает контейнер на основе указанного образа, а docker start используется для управления сетевыми настройками контейнера  
- docker run используется для запуска контейнера в фоновом режиме, а docker start запускает контейнер в интерактивном режиме

## Вопрос 4

- Загружает файлы из URL в образ с внешних источников без дополнительной обработки  
- Копирует файлы и директории из хоста в образ, поддерживая распаковку архивов  
- Открывает указанный порт для дополнительного доступа к контейнеру  
- Добавляет и устанавливает необходимые переменные окружения во время сборки образа  
- Добавляет команды во время сборки образа, аналогично RUN

## Вопрос 7  
У вас есть два файла Docker Compose. В одном из них в секции services описан только один сервис, а в другом несколько, объединённых в сеть.  
Чем отличаются подходы использования этих двух секций services?  

- Первый файл обеспечивает связь между всеми контейнерами, второй ограничивает связь только между конкретными контейнерами  
- Первый файл запускает контейнеры из разных сетей, второй создает одну сеть для каждого контейнера  
- Первый файл удобен для тестирования отдельных компонентов, второй обеспечивает полноценное многоконтейнерное окружение  
- Первый файл запускает только базовые контейнеры, второй поддерживает исключительно сложные многоконтейнерные приложения  
- Первый файл требует меньше системных ресурсов, второй автоматически масштабирует контейнеры

## Вопрос 8  
Вы настраиваете среду разработки для команды, работающей над приложением на основе микросервисной архитектуры. Приложение включает три контейнера: API-сервер, кэш и база данных. При этом:  
1. Все три контейнера должны запускаться и останавливаться одновременно.  
2. Кэш должен использовать постоянное хранилище, чтобы данные сохранялись между перезапусками.  
3. API-сервер должен автоматически перезапускаться при сбое.  
Как настроить Docker Compose для реализации этих требований?  

- Использовать параметр build в Docker Compose файле для всех контейнеров, чтобы обеспечить их одновременный синхронный запуск  
- Создать файл с секциями services для всех контейнеров, добавить volumes для кэша и установить параметр restart: always для API-сервера  
- Использовать команду docker-compose up для запуска контейнеров и команду docker-compose down для остановки контейнеров  
- Создать Docker Compose файл с секцией networks и подключить все эти три контейнера к одной сети  
- Добавить параметр depends_on для кэша и базы данных, чтобы запускать эти три контейнера перед API-сервером

## Вопрос 16  
Какое из следующих утверждений верно в контексте настройки частного Docker Registry?  

- Использование Docker Compose для запуска реестра упрощает управление конфигурацией  
- Настройка переменных среды через docker run позволяет гибко управлять параметрами реестра  
- Настройка TLS необходима для обеспечения безопасного соединения с реестром  
- Аутентификация через docker login обязательна для доступа к частному реестру  
- Режим отладки позволяет отслеживать все действия пользователей для аудита
